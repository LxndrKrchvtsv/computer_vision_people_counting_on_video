## Часть 1 — Генерация стадийных картинок

### Что уже реализовано

В `src/pipeline.py` при `save_stages=True` каждые N кадров (по умолчанию 300)
в папку `data/output/` сохраняются 6 изображений на каждый checkpoint-кадр:

| # | Файл | Стадия пайплайна |
|---|------|------------------|
| 1 | `XXXXX_1_original.jpg` | Исходный кадр |
| 2 | `XXXXX_2_enhanced.jpg` | После CLAHE + Gaussian denoise |
| 3 | `XXXXX_3_raw_mask.jpg` | Маска MOG2 (сегментация фона) |
| 4 | `XXXXX_4_clean_mask.jpg` | Очищенная маска (морфология + filter_components) |
| 5 | `XXXXX_5_detection.jpg` | Рамки детекций на исходном кадре |
| 6 | `XXXXX_6_final.jpg` | Финальный кадр (трекинг + счётчик IN/OUT) |

### Как сгенерировать

```powershell
# Из корня проекта C:\Users\akric\PycharmProjects\cv_project
$env:PYTHONIOENCODING='utf-8'
.venv\Scripts\python.exe main.py --video data/test.mp4
```

Это запустит **классический** режим (MOG2 + CentroidTracker) с `save_stages=True`.
Время выполнения ~4 мин на CPU. Картинки появятся в `data/output/`.

Для отчёта выбрать кадр, где хорошо видно людей (рекомендую **00300** или **00600**).

### Важно

- Стадийные картинки генерируются только для **классического** режима.
  В YOLO-ветке стоит `continue` до блока `save_stage` — промежуточных стадий нет.
- Для YOLO-режима финальный результат — видео `data/output/result.mp4`.
  Можно вручную сделать скриншот из видео для отчёта (любой кадр с треками).
- Запуск YOLO-режима (если нужен скриншот):
  ```powershell
  .venv\Scripts\python.exe main.py --video data/test.mp4 --yolo
  ```

## Часть 3 — Структура PDF-отчёта

### Рекомендуемый порядок разделов

| # | Раздел | Содержание |
|---|--------|------------|
| 1 | Введение | Постановка задачи: подсчёт людей и направление движения на видео пешеходного перехода (тема 14) |
| 2 | Пайплайн | Схема `image → enhance → segment → clean → detect → decide`. Краткое описание каждой стадии |
| 3 | Стадии (картинки) | 6 картинок с одного кадра + подпись к каждой |
| 4 | Методы | Два подхода: Классика (MOG2 + CentroidTracker) и YOLO (YOLOv8s + ByteTrack + LineCounter) |
| 5 | Сравнение методов | Таблица бенчмарка (IN/OUT/IDs/Time) + ground truth |
| 6 | Failure cases | Проблемы классики: слияние силуэтов, ложные срабатывания (зебра, листья, стены, машины), мерцание боксов |
| 7 | Выводы | Итоги сравнения (см. ниже) |
| 8 | Contribution | Кто что делал |

### Ключевые выводы для раздела 7

- YOLO даёт **5× меньше текучки ID** (193 vs 1041) → значительно стабильнее.
- Оба метода недосчитывают ~22% (YOLO: IN −22%, OUT −22%; классика: IN −24%, OUT −16%).
- Причины недосчёта разные:
  - Классика — MOG2 реагирует на любое движение (не только людей), трекер теряет ID.
  - YOLO — пропуски детекции на дальнем плане (мелкие фигуры).
- YOLO медленнее на CPU (~323s vs ~249s) из-за inference нейросети.
- Классика фундаментально непригодна для плотной толпы (failure case для отчёта).

### Failure cases — что показать

1. **Слияние силуэтов** — на маске `3_raw_mask` видно, как несколько человек
   сливаются в один блоб. Один бокс на нескольких людей.
2. **Ложные срабатывания** — зебра перехода, листья деревьев, стены зданий
   создают «призрачные» детекции на `4_clean_mask`.
3. **Мерцание ID** — на видео `result.mp4` (классика) видно, как один человек
   получает разные ID на последовательных кадрах. ID доходят до 1000+.
