# Фаза 1 — Переработка трекинга и счёта

Цель фазы: убрать мелькание квадратов и сделать подсчёт людей достоверным.

Формат работы: на каждом шаге агент даёт **теорию** (зачем) + **практику**
(что писать, примеры кода как образец), затем **ты пишешь код сам**. После шага —
проверка прогоном на `data/test.mp4`, фиксация результата в `docs/PHASES.md`.

---

## Почему мелькание — это проблема ТРЕКИНГА, а не детекции

Детектор отвечает на вопрос «где люди на ЭТОМ кадре». Трекер отвечает «это тот же
человек, что и на прошлом кадре». Боксы «теряются и мелькают», потому что трекер
не сопоставляет объекты между кадрами и плодит новые ID. Поэтому Фаза 1 чинит
два слоя: (1) источник стабильных ID и (2) логику подсчёта поверх этих ID.

## Шаги фазы 1

### Шаг 1.1 — Теория трекинга (понимание)
- Что такое tracking-by-detection, ID switch, окклюзия, motion model.
- Почему текущий `CentroidTracker` ломается на плотной толпе.
- Что такое ByteTrack и почему он решает проблему.
- **Артефакт:** понимание, изменений кода нет.

### Шаг 1.2 — Стабильные ID через ByteTrack
- Перевести детекцию людей на `model.track(..., persist=True, tracker="bytetrack.yaml")`
  вместо `model()` + самописный трекер.
- Источник ID — теперь сам ByteTrack (поле `boxes.id`).
- **Файлы:** `src/detect_yolo.py` (новая функция трекинга), `src/pipeline.py`.
- **Критерий приёмки:** на прогоне ID растут медленно и стабильно (а не до #1300
  к 600-му кадру); боксы держатся на людях, не мелькают.

### Шаг 1.3 — Переписать логику подсчёта (`decision.py`)
- Счёт по стабильным track ID: каждый ID пересекает линию и считается ОДИН раз.
- Хранить для каждого ID историю центроидов (последние N позиций).
- Определять пересечение по переходу центра через линию между кадрами.
- **Файл:** `src/decision.py`.
- **Критерий приёмки:** один человек = +1 к счётчику (без двойного счёта при
  кратковременной потере/смене позиции).

### Шаг 1.4 — Направление и flow
- Линия с ориентацией (горизонтальная/наклонная) + знак перемещения вдоль нормали
  → IN/OUT.
- Базовая оценка flow (нетто = IN − OUT, и/или активные треки) для требования темы
  «People Counting **& Flow**».
- **Файлы:** `src/decision.py`, `src/visualize.py` (отрисовка линии/направления).
- **Критерий приёмки:** счётчики IN/OUT соответствуют видимому направлению на демо.

### Шаг 1.5 — Сглаживание у линии и анти-дребезг
- Засчитывать пересечение только при наличии ≥ N последних позиций трека.
- Гистерезис: небольшая «мёртвая зона» вокруг линии, чтобы дрожание центра у
  линии не давало повторных срабатываний.
- **Файл:** `src/decision.py`.
- **Критерий приёмки:** нет всплесков счёта от людей, стоящих на линии.

### Шаг 1.6 — Проверка и фиксация
- Прогон на `data/test.mp4`, сохранить result.mp4 и числа.
- Зафиксировать новые IN/OUT в `docs/PHASES.md` рядом с baseline.
- Визуально сверить на коротком отрезке (заготовка для Фазы 5 — валидация).

## Архитектурное решение фазы

- **Основной счёт** строится на YOLO + ByteTrack (надёжно, стабильные ID).
- Логика подсчёта в `decision.py` пишется так, чтобы НЕ зависеть от источника ID —
  на вход принимает детекции с уже присвоенными ID. Это позволит позже (Фаза 4)
  подать туда и «классические» детекции для сравнения методов.
- Самописный `CentroidTracker` не удаляем: пользователь выносит его в отдельный
  файл с пометкой (напр. `src/legacy/centroid_tracker.py`) для бенчмарка в Фазе 4.
